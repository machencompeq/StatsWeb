## 1. 核心概念：人看得懂 ↔ 電腦看得懂

輸入層要解決的，就是「人」和「電腦」對符號的認知落差：

- **學生看到的**：熟悉的數學符號（x̄、σ²、χ²、H₀…），跟課本、手寫一模一樣
- **電腦存的**：一段標準編碼（**LaTeX**），可被程式精準解析、未來能直接送批改

```mermaid
flowchart LR
    A["👩‍🎓 學生點按鈕<br/>看到熟悉符號<br/>例：x̄"] --> B["⌨️ 系統插入電腦編碼<br/>例：\\bar{x}"]
    B --> C["🖥️ 畫面即時渲染<br/>KaTeX 顯示回 x̄"]
    C --> D["💾 送出時保存的是編碼<br/>機器可讀、可批改"]

    classDef stu fill:#E3F2FD,stroke:#1565C0,color:#0D47A1;
    classDef sys fill:#E8F5E9,stroke:#2E7D32,color:#1B5E20;
    classDef save fill:#FFF3E0,stroke:#E65100,color:#BF360C;
    class A stu;
    class B,C sys;
    class D save;
```

> **為什麼選 LaTeX 當電腦格式**：純文字存不了上下標、分數、希臘字母；LaTeX 能精準還原數學式、用 KaTeX/MathJax 渲染成漂亮考卷、且是業界標準的機器可讀格式——未來接批改時可直接餵入，不必重做。學生**完全不需要懂 LaTeX**，他們只按符號鈕，編碼由系統在背後處理。

## 2. 三個畫面狀態（同一個符號的三種樣貌）

| 階段 | 學生看到 | 系統內部 | 說明 |
|------|----------|----------|------|
| ① 按鈕上 | x̄ | — | 學生只認得這個 |
| ② 作答區渲染 | x̄ | `\bar{x}` | 即時渲染，所見即所得 |
| ③ 送出保存 | x̄（可重新渲染） | `\bar{x}` | 資料庫存的是編碼 |

## 4. 過程 vs 關鍵答案的分工

過程／推導用上面的符號面板手打；但**最終要精準批改的數字**走獨立結構化欄位，不經符號面板，從根本避免輸入歧義：

| 欄位 | 輸入元件 | 電腦儲存 |
|------|----------|----------|
| H₀ / H₁ | 下拉 + 少量符號鈕 | LaTeX |
| 檢定統計量 | 純數字欄 | 數值 |
| p 值 | 純數字欄 | 數值 |
| 自由度 | 純數字欄 | 數值 |
| 結論（拒絕 / 不拒絕 H₀） | 下拉選單 | 列舉值 |

## 5. 技術選型（皆免費、開源）

| 需求 | 建議工具 | 說明 |
|------|----------|------|
| 數學輸入框 + 虛擬鍵盤 | **MathLive**（開源） | 自帶可自訂虛擬鍵盤、輸出 LaTeX，省去自製面板；本規格的符號表即用來設定它要顯示哪些鍵 |
| 純渲染（顯示考卷/題目） | **KaTeX** 或 MathJax | 把 LaTeX 渲染成漂亮數學式，速度快 |

> 取捨：自製按鈕面板可完全客製外觀，但工程量大；MathLive 直接給你輸入框＋鍵盤＋LaTeX 輸出，建議優先評估，再用本規格的符號清單去設定它顯示哪些符號。
